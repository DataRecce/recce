# Deep Dive: DRC-3039 — Add export options for row limit handling

## Issue Summary

The UI preview table is limited to 2,000 rows (correct default), but CSV/TSV/Excel exports currently export only that preview subset. Exports should download ALL rows from the query, display row counts on download buttons, and warn for large exports (>100k rows).

## Classification
**Type:** Feature
**Priority:** Medium
**Labels:** None
**Assignee:** Jared Scott

## Current State

### Backend (Python)
- `recce/tasks/query.py:14` defines `QUERY_LIMIT = 2000`
- `QueryMixin.execute_sql_with_limit()` applies this limit at query execution time — the backend only fetches 2,001 rows (2,000 + 1 to detect "more")
- `DataFrame` model has `limit` and `more` fields that track the cap
- **Key insight:** The limit is applied at the DB query level, not just the UI. To export all rows, we need a separate backend endpoint or parameter to fetch without the limit.

### Frontend (TypeScript)
- `useCSVExport` hook (L43-261) extracts data from `run.result` — which already has the 2,000 row limit baked in
- `extractCSVData()` in `utils/csv/extractors.ts` simply reads the DataFrame from the run result
- Export menu lives in `RunResultPane.tsx` — `DefaultExportMenu` and `DefaultShareMenu` components render download buttons as simple text labels (no row counts)
- No confirmation dialog exists for large exports

## Risks and Open Questions

1. **Backend data fetching:** The 2,000 row limit is enforced at the SQL execution level. Exporting all rows requires either:
   - A new API endpoint that re-runs the query without limit
   - A streaming/pagination endpoint for large results
   - Passing a "no limit" flag to the existing query execution
2. **Memory pressure:** Fetching millions of rows into the browser could crash the tab. Need to consider streaming or server-side file generation.
3. **Total row count:** To show "Download as CSV (1.2M rows)" we need the total row count BEFORE the user clicks download. This may require a separate `COUNT(*)` query or storing the count from the original query.
4. **Which run types?** The issue focuses on query/query_diff, but the export menu applies to all run types (profile_diff, row_count_diff, value_diff, top_k_diff). Scope question: all types or just query?

## Proposed Approach

### Architecture Decision: Server-side export

Given that exports can be millions of rows, generating files in the browser from full datasets is impractical. The approach should be:

1. **New backend API endpoint** (`/api/export`) that:
   - Re-runs the query without the 2,000 row limit
   - Streams results directly to a CSV/TSV/Excel file response
   - Returns the file as a download (not loading all data into browser memory)

2. **Row count endpoint or metadata** — either:
   - Add a `total_row_count` field to the existing query result (run a `SELECT COUNT(*)` wrapper)
   - Or use the `more` flag + separate count query on demand

3. **Frontend changes:**
   - Update download buttons to show row counts
   - Add confirmation dialog for >100k rows
   - Download buttons trigger the new backend endpoint instead of client-side file generation
